## 系统简介

**NexusRoute** 是一个基于 Ubuntu + Xray 的工业级透明代理网关系统，专为需要**多用户隔离**、**防IP泄露**、**动态切换代理链路**的场景设计。

### 核心用途

1. **隐私保护**：确保内网设备的真实IP永不泄露
2. **多用户管理**：支持最多200个设备，每个设备独立配置代理路径
3. **动态级联**：用户可随时切换1-3跳代理链，无需重启网关
4. **自动化管理**：新设备自动发现，管理员审批后即可使用

### 详细对比分析

#### 1. 架构设计理念

**OpenWrt 方案**：
```
路由器硬件
  └─ OpenWrt 系统
      └─ PassWall/SSR+ 插件
          └─ 所有设备共享同一套代理配置
```
- **优势**：轻量级，适合嵌入式设备
- **劣势**：所有设备使用相同的代理路径，无法个性化

**NexusRoute 方案**：
```
服务器/虚拟机
  └─ Ubuntu Server
      └─ 多个独立 Xray 实例
          ├─ 用户A: 香港→日本
          ├─ 用户B: 美国→英国→德国
          └─ 用户C: 新加坡
```
- **优势**：每个用户独立配置，互不影响
- **劣势**：资源占用较高

#### 2. 使用场景对比

| 场景 | OpenWrt 更优 | NexusRoute 更优 |
|------|-------------|----------------|
| 家庭单一代理需求 | ✅ | ❌ |
| 多人共享同一代理 | ✅ | ❌ |
| 低功耗24小时运行 | ✅ | ❌ |
| 多用户独立配置 | ❌ | ✅ |
| 渗透测试/安全研究 | ❌ | ✅ |
| 频繁切换代理链路 | ❌ | ✅ |
| 设备接入审批 | ❌ | ✅ |
| 防IP泄露要求极高 | ⚠️ | ✅ |
| 虚拟化环境部署 | ❌ | ✅ |
| 临时测试环境 | ❌ | ✅ |

#### 3. 技术实现对比

**代理切换机制**：

OpenWrt：
```
1. 管理员登录 LuCI 界面
2. 修改 PassWall 配置
3. 重启服务 (所有用户断网 5-10秒)
4. 所有设备使用新配置
```

NexusRoute：
```
1. 用户访问 http://192.168.100.1/
2. 选择新的代理节点
3. 点击"应用配置"
4. 仅该用户重启 (3秒)，其他用户不受影响
```

**防IP泄露机制**：

OpenWrt：
```
依赖正确配置：
- iptables 规则
- 防火墙设置
- 路由表配置
- DNS 劫持

风险：配置错误可能导致泄露
```

NexusRoute：
```
强制机制：
- TPROXY 强制劫持
- FORWARD 链默认 DROP
- MAC-IP 双重验证
- 代理停止 = 完全断网

保证：宁可断网，绝不泄露
```

#### 4. 性能对比

**OpenWrt**：
- ✅ 吞吐量高 (C语言实现，接近线速)
- ✅ 延迟低 (内核态处理)
- ✅ 资源占用小 (64MB内存可运行)
- ⚠️ 并发连接数受硬件限制

**NexusRoute**：
- ⚠️ 吞吐量中等 (Node.js + Xray)
- ⚠️ 延迟略高 (用户态处理)
- ❌ 资源占用大 (2GB内存起步)
- ✅ 并发连接数高 (服务器级硬件)

### NexusRoute 的核心优势

   - 测试环境需要模拟不同地区

2. **安全研究/渗透测试**
   - 需要频繁切换代理链路
   - 需要多层级联隐藏来源
   - 需要严格防止IP泄露

3. **虚拟化环境**
   - Hyper-V/VMware/KVM虚拟机
   - 容器化部署
   - 临时测试环境

4. **设备管控需求**
   - 需要审批新设备接入
   - 需要MAC-IP绑定
   - 需要随时切断特定设备

5. **隐私保护优先**
   - 无日志设计
   - Kill Switch强制断网
   - 多层代理增强匿名性

#### ❌ NexusRoute 的局限性

1. **不适合的场景**
   - 家庭单一代理需求（OpenWrt更简单）
   - 低功耗设备（树莓派3B及以下）

   - 只专注代理功能
   - 无广告过滤
   - 无QoS流控
   - 无DDNS等路由器功能

## 核心特性

### 🔒 安全特性

- **Kill Switch（断网防漏油）**：代理断开时自动切断网络，防止真实IP泄露
- **MAC-IP双重绑定**：防止IP欺骗攻击
- **TPROXY透明代理**：应用层无感知，无需配置客户端
- **无日志设计**：不记录访问日志，保护隐私

### 👥 多用户管理

- **独立实例**：每个用户独立的Xray进程，互不影响
- **自动发现**：新设备接入自动检测，管理员审批后分配固定IP
- **用户隔离**：A用户切换节点不影响B用户的连接

### 🔗 动态级联

- **1-3跳代理链**：支持单跳直连或多跳级联
- **实时切换**：用户通过Web界面随时更改代理路径
- **多协议支持**：VMess、VLESS、Trojan、Shadowsocks、Socks5

### 🎨 易用性

- **Web管理界面**：现代化UI，无需命令行操作
- **一键安装**：自动化部署脚本，10分钟完成安装
- **自动配置**：设备审批后自动配置所有组件

## 部署场景

NexusRoute 支持多种部署场景，只要满足以下条件即可：

**硬件要求**：
- 一台运行 Ubuntu Server 22.04 LTS 的设备
- 至少 2GB 内存
- 至少 10GB 磁盘空间
- **两个网络接口**（一个连外网，一个连内网）

**网络要求**：
- 外网接口能访问互联网
- 内网接口连接需要代理的设备
- 内网设备只能通过网关访问外网（不能有其他出口）

### 数据流向详解

#### 1. 设备接入流程

```
新设备连接到 eth1
    │
    ├─► DHCP 请求 → dnsmasq 分配临时IP (192.168.100.50-99)
    │
    ├─► 后台监控检测到新MAC地址
    │
    ├─► 插入 pending_devices 表
    │
    ├─► 管理员在后台看到"待审批设备"
    │
    ├─► 管理员点击"批准"
    │
    ├─► 系统执行：
    │   ├─ 基于MAC哈希生成永久IP (192.168.100.10-209)
    │   ├─ 创建用户记录 (user1, user2...)
    │   ├─ 分配独立端口 (12345, 12346...)
    │   ├─ 写入 dnsmasq 静态绑定
    │   ├─ 创建 Xray systemd 服务
    │   ├─ 添加 iptables TPROXY 规则
    │   └─ 重启 dnsmasq
    │
    └─► 设备重新获取IP，获得永久IP和上网权限
```

#### 2. 网络流量路径

```
用户设备 (192.168.100.10) 访问 google.com
    │
    ├─► DNS 查询 (UDP 53)
    │   └─► dnsmasq (eth1:53)
    │       └─► 转发到 127.0.0.1:5353
    │           └─► Xray DNS 入站
    │               └─► 通过代理链查询
    │                   └─► 返回 IP 地址
    │
    └─► HTTP/HTTPS 请求 (TCP 443)
        │
        ├─► iptables PREROUTING 链
        │   ├─ 检查源IP: 192.168.100.10 ✓
        │   ├─ 检查MAC: aa:bb:cc:dd:ee:01 ✓
        │   ├─ 打标记: mark 0x1
        │   └─ TPROXY 到端口 12345
        │
        ├─► ip rule 路由到 table 100
        │
        ├─► Xray-user1 实例 (端口12345)
        │   │
        │   ├─► 读取配置: /usr/local/etc/xray/config-user1.json
        │   │
        │   └─► 级联代理链
        │       │
        │       ├─► 第1跳: 香港节点 (VMess over WebSocket+TLS)
        │       │   └─► 连接到 hk.example.com:443
        │       │
        │       ├─► 第2跳: 日本节点 (Trojan over TLS)
        │       │   └─► 通过第1跳连接到 jp.example.com:443
        │       │
        │       └─► 第3跳: 美国节点 (VLESS over TLS)
        │           └─► 通过第2跳连接到 us.example.com:443
        │
        └─► 从美国节点出口访问 google.com
            └─► 返回数据原路返回
```

#### 4. Kill Switch 防漏油机制

```
┌─────────────────────────────────────────────────────────────┐
│                      正常工作状态                             │
│                                                               │
│  内网设备 → TPROXY → Xray (运行中) → 代理节点 → 互联网      │
│                                                               │
│  结果: ✅ 正常访问，真实IP隐藏                               │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    代理服务停止状态                           │
│                                                               │
│  内网设备 → TPROXY → Xray (已停止) → ❌ 连接失败            │
│      │                                                        │
│      └─► 尝试直接转发 → FORWARD 链 (policy DROP) → ❌ 拒绝  │
│                                                               │
│  结果: ✅ 完全断网，真实IP不泄露                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  Ubuntu 网关自身流量                          │
│                                                               │
│  网关 apt update → OUTPUT 链 → eth0 → 互联网                │
│                                                               │
│  结果: ✅ 网关自身可以访问外网（不经过TPROXY）               │
└─────────────────────────────────────────────────────────────┘
```

### 安全边界

```
┌─────────────────────────────────────────────────────────────┐
│                        安全分层                               │
│                                                               │
│  第1层: MAC-IP 绑定                                          │
│  ├─ 防止 IP 欺骗攻击                                         │
│  └─ 如果 IP 和 MAC 不匹配 → DROP                            │
│                                                               │
│  第2层: TPROXY 强制劫持                                      │
│  ├─ 所有 TCP/UDP 流量必须经过 Xray                          │
│  └─ 应用层无法绕过                                           │
│                                                               │
│  第3层: FORWARD DROP                                         │
│  ├─ 默认拒绝所有转发                                         │
│  └─ 即使 TPROXY 失效也不会泄露                              │
│                                                               │
│  第4层: 无日志设计                                           │
│  ├─ Xray 日志级别: warning                                  │
│  ├─ 不记录访问历史                                           │
│  └─ 不记录流量统计                                           │
│                                                               │
│  第5层: 设备审批                                             │
│  ├─ 新设备必须管理员批准                                     │
│  └─ 未批准设备无法上网                                       │
└─────────────────────────────────────────────────────────────┘
```

## 使用教程

### 1. 添加代理节点

访问 `http://192.168.100.1/admin` → 登录 → 节点管理 → 添加节点

**VMess 示例**：
```
节点名称: 香港01
协议: VMess
服务器地址: hk.example.com
端口: 443
UUID: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
传输协议: WebSocket
WebSocket Path: /vmess
TLS: TLS
SNI: hk.example.com
```

**Trojan 示例**：
```
节点名称: 日本02
协议: Trojan
服务器地址: jp.example.com
端口: 443
密码: your-password
传输协议: TCP
TLS: TLS
```

### 2. 连接设备

将设备连接到内网接口（eth1），设备会自动获取临时IP（192.168.100.50-99）。

### 3. 审批设备

管理后台 → 设备审批 → 看到新设备 → 点击"批准"

系统会自动分配永久IP（192.168.100.10-209）并创建用户。

### 4. 配置代理路径

设备访问 `http://192.168.100.1/` → 选择1-3跳节点 → 应用配置

**出口IP说明**：
- 选1跳：出口是第1跳的IP
- 选2跳：出口是第2跳的IP
- 选3跳：出口是第3跳的IP

## 防漏油测试

### 测试步骤

1. **停止代理服务**

```bash
sudo systemctl stop 'xray-user*'
```

2. **在内网设备测试**

```bash
ping 8.8.8.8          # 应该超时
curl https://ipinfo.io # 应该失败
```

3. **预期结果**

✅ 正确：完全无法访问互联网  
❌ 错误：仍能访问（表示有漏油风险）

4. **恢复服务**

```bash
sudo systemctl start 'xray-user*'
```

## 故障排查

### 设备无法获取IP

```bash
sudo systemctl status dnsmasq
sudo journalctl -u dnsmasq -n 50
```

### 设备无法上网

```bash
sudo iptables -t mangle -L -n -v
sudo systemctl status xray-user1
```

### 管理后台无法访问

```bash
sudo systemctl status nexusroute
sudo journalctl -u nexusroute -n 50
```

## 常见问题

**Q: 可以在非Hyper-V环境使用吗？**  
A: 可以！只要有两个网络接口即可，支持物理机、VMware、VirtualBox、KVM、Docker等。

**Q: 可以用树莓派部署吗？**  
A: 可以，但需要USB网卡作为第二个接口。树莓派4B推荐。

**Q: 支持WiFi作为内网接口吗？**  
A: 理论可以，但需要配置WiFi AP模式，建议使用有线网卡。

**Q: 最多支持多少设备？**  
A: 200个设备（IP池限制）。

**Q: 出口IP是哪个节点？**  
A: 最后一跳的节点IP。例如选2跳，出口就是第2跳。

**Q: 如何备份配置？**  
A: 备份 `/opt/nexusroute/db.sqlite` 文件即可。
